# Jira ↔ GitHub Issue Sync — Design Spec

**Date:** 2026-03-31
**Status:** Approved

---

## Overview

A standalone GitHub Action distributed on the GitHub Marketplace that synchronises GitHub issues with Jira tasks. GitHub is the source of truth. When a GitHub issue is opened, a Jira Task is created. When the issue is closed, the Jira task is transitioned to Done.

---

## Inputs

| Input | Required | Description |
|---|---|---|
| `jira-base-url` | yes | Atlassian domain, e.g. `https://your-org.atlassian.net` |
| `jira-user-email` | yes | Email of the Jira user associated with the API token |
| `jira-api-token` | yes | Jira API token (passed as a GitHub secret) |
| `jira-project-key` | yes | Key of the Jira project to create tasks in, e.g. `MYPROJECT` |
| `jira-epic-key` | yes | Key of the Jira Epic to link new tasks to, e.g. `MYPROJECT-42` |

All secrets (`jira-base-url`, `jira-user-email`, `jira-api-token`) should be stored as GitHub Actions secrets and referenced via `${{ secrets.* }}`.

---

## Trigger Flow

Users add this workflow to their repository:

```yaml
on:
  issues:
    types: [opened, closed]

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: <owner>/jira-issue-sync@v1
        with:
          jira-base-url: ${{ secrets.JIRA_BASE_URL }}
          jira-user-email: ${{ secrets.JIRA_USER_EMAIL }}
          jira-api-token: ${{ secrets.JIRA_API_TOKEN }}
          jira-project-key: 'MYPROJECT'
          jira-epic-key: 'MYPROJECT-42'
```

### When a GitHub issue is opened

1. Read the issue title, body, and URL from the GitHub event payload.
2. Create a Jira Task in the specified project, linked to the specified Epic. The task summary should be the issue title, and the description should include the issue body and a link to the GitHub issue.
3. Post a comment on the GitHub issue: "Jira task created: [`PROJ-123`](https://your-org.atlassian.net/browse/PROJ-123)".

### When a GitHub issue is closed

1. Fetch all comments on the GitHub issue.
2. Find the bot comment matching the known pattern (e.g. `"Jira task created: ..."`).
3. Extract the Jira issue key from the URL in that comment.
4. Transition the Jira task to `"Done"`.
5. If no matching comment is found: log a warning via `core.warning` **and** post a comment on the GitHub issue explaining that the linked Jira task could not be found and the sync was skipped.

## Project Structure

```
jira-issue-sync/
├── action.yml              # Action metadata & input declarations
├── src/
│   ├── index.ts            # Entry point — reads inputs, dispatches to handlers
│   ├── handlers/
│   │   ├── opened.ts       # Logic for issue-opened event
│   │   └── closed.ts       # Logic for issue-closed event
│   ├── jira.ts             # Jira API client (create task, transition, link to epic)
│   └── github.ts           # GitHub API helpers (post comment, fetch comments)
├── dist/                   # Compiled bundle (committed, generated by ncc)
├── .github/
│   └── workflows/
│       ├── ci.yml          # Lint + test on push / PR
│       └── release.yml     # Build dist/ and publish on tag push
├── .husky/                 # Pre-commit hooks
├── package.json
├── tsconfig.json
├── .eslintrc
├── .gitignore
└── README.md
```

### Responsibilities

- **`index.ts`** — reads `context.payload.action` (`opened` vs `closed`) and delegates to the appropriate handler. No business logic.
- **`handlers/opened.ts`** — orchestrates: read issue data → create Jira task → post GitHub comment.
- **`handlers/closed.ts`** — orchestrates: fetch comments → find Jira link → transition task (or warn + comment).
- **`jira.ts`** — thin wrapper around the Jira REST API. Exposes: `createTask`, `linkToEpic`, `transitionToDone`.
- **`github.ts`** — thin wrapper around the GitHub REST API (via `@actions/github`). Exposes: `postComment`, `fetchComments`.

### Code Style

- **Simplicity and expressiveness first.** Small, single-purpose functions.
- **Long, descriptive names over abbreviations.** (e.g. `jiraIssueKey` not `key`, `githubIssueUrl` not `url`).
- **No silent failures.** Every error surfaces a clear message.
- Follow *The Pragmatic Programmer* principles: orthogonality, DRY, clear intent.

## Error Handling

| Scenario | Behaviour |
|---|---|
| Jira task creation fails | `core.setFailed` with HTTP status + response body |
| Jira transition fails | `core.setFailed` with HTTP status + response body |
| No Jira link comment found on close | `core.warning` + post a comment on the GitHub issue |
| Unknown `action` value in event payload | `core.setFailed` with descriptive message |

## MVP Constraints & Future Work

The following are intentionally deferred to keep the MVP focused:

| Deferred | Notes |
|---|---|
| Configurable Jira issue type | Hardcoded to `Task` for now |
| Configurable created status | Jira project default for now |
| Configurable closed status | Hardcoded to `Done` for now |

However, the code structure should allow for these to be added later without major refactoring. For example, the Jira API client can be designed to accept issue type and status as parameters, even if the handlers currently pass hardcoded values. They can also be added as inputs with default values, so they are optional for MVP users but available for advanced use cases.
